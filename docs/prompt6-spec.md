# Prompt 6 設計仕様: 検索 / フィルタ最小

このドキュメントは `/works` 一覧ページに最小限の検索・フィルタを導入する Prompt 6 の設計仕様をまとめたものです。

関連ドキュメント:

- [Artli の前提思想と実装判断基準](./philosophy.md)
- [ロードマップと Prompt 進捗](./roadmap.md)

---

## 1. 目的

**作品発見性の向上**。取得効率化や調達最適化ではありません。

具体的には以下を狙います。

- 公開作品を見つけやすくする
- 作家に出会いやすくする
- 作家の公開方針を軽く理解しながら一覧を見られるようにする

Artli は作家主体のプロダクトです（詳しくは [philosophy.md](./philosophy.md)）。検索機能もその前提思想を崩さない範囲で実装します。

---

## 2. 対象範囲

- 対象ページ: **`/works` 一覧ページのみ**
- 対象データ: **public 作品のみ**
- 対象外: `/artists` 側の検索拡張は**含めない**

---

## 3. 検索対象（3 つに限定）

検索バーに入力された文字列は、以下の 3 つに対してのみマッチさせます。

- 作品タイトル
- タグ
- 作家表示名

### 検索対象に含めないもの

- 作品の説明文
- スラッグ / ID
- ライセンス値そのもの（例: `commercial=allowed` のような内部値）

---

## 4. フィルタ対象（4 項目）

UI 上に公開するフィルタは以下の 4 項目のみです。

- `trainingType`
- `adult`
- `commercial`
- `consult` 関連

### フィルタに含めないもの

- `redistribution`
- `priceJpy`（価格）
- 取得可否ベースの条件
- タグや作家の個別フィルタ（タグ一覧画面 / 作家ページでの絞り込みは別課題）

---

## 5. `consult` の扱い

`consult`（要相談）の扱いは、Artli の前提思想上とくに重要です。

- `consult` は「即時取得可能性」ではなく、**作家の公開方針のひとつ**として扱う
- **デフォルトでは要相談作品も含めて表示**する
- フィルタは「要相談作品を除く」程度に留める
- 「即時取得可能のみ」「取得できる作品のみ」のような **developer 主語の表現は避ける**

---

## 6. UI 方針

### 基本構成

- 画面上部に**検索バー**と、その下あたりに**軽量なフィルタ列**
- 主役はあくまで **一覧カード**
- 検索は主導線、フィルタは補助
- **サイドバー型にしない**
- **管理画面 / 業務 UI のようにしない**

### 避けたい方向

- 左サイドバーに多数のフィルタを並べる構成
- 価格や取得可否を前面に出した導線
- ライセンス条件を比較表のように見せる構成

---

## 7. 一覧カードの方針

一覧カードは「作品と作家に出会う」体験を壊さない情報密度に留めます。

- **主役**: サムネイル / タイトル / 作家名
- **可能なら活かす**: 作家アイコン
- **軽く表示してよい**: `trainingType`、控えめな `consult` 補足
- **前面に出さない**: `commercial`, `adult`, `redistribution`, `priceJpy`, 取得可否, 取得済み表示

完成形で見たときに「作品と作家が主役」に見えることを最優先します。

---

## 8. 実装アーキテクチャ方針

### ルーティングとレンダリング

- `/works` は**引き続き SSR / Server Component ベース**
- `searchParams` を page 側で受ける（現在の `src/app/works/page.tsx` を拡張する形）

### データ取得

- 一覧取得ロジックは既存の `src/lib/queries.ts` の `getPublicWorks()` を**拡張**する
- Prisma の `where` 構築は helper 側で行う
- **新しい専用検索 API は作らない**

### クエリ構築ルール

- ベース条件は常に **public 作品のみ**
- 検索語はタイトル・タグ・作家表示名に対する **OR 条件**
- フィルタは **AND 条件**で追加
- `consult` 判定は helper 内で吸収する（呼び出し側に consult ロジックを露出しない）

---

## 9. URL と状態保持

検索・フィルタの状態は **URL クエリで保持**します。ページリロード・戻る・リンク共有で同じ状態が再現できます。

URL クエリで保持する対象:

- 検索語
- `trainingType`
- `adult`
- `commercial`
- `consult`

未指定時は**絞り込みなし**として扱います。

---

## 10. 検索結果表示の方針

- ページ見出しは **Works のまま維持**する（「検索結果」などに差し替えない）
- **結果件数は控えめに表示**する
- 現在の条件は**軽いチップや補助表示**で示す
- **0 件時は柔らかい文言**を使う
  - 「条件に一致するデータがありません」のような事務的な文言は避ける
- **「すべてクリア」導線は控えめ**に置く

---

## 11. Prompt 6 でやらないこと

スコープを守るため、以下は Prompt 6 では**明示的にやりません**。

- `/artists` 側の検索拡張
- 高度な全文検索（ステミング・形態素解析・スコアリングなど）
- 専用検索 API の新設
- ソート機能追加（新着順・人気順など）
- 価格検索 / 価格の前面表示
- 取得可否中心の検索導線
- 一覧カードでのライセンス情報の大量表示
- 相談フロー本体の実装
- Prisma schema / License モデルの再設計
- レコメンドや保存検索などの発展機能

---

## 12. 迷ったときの優先順位

Prompt 6 の実装中に判断に迷ったら、以下の順で優先します（[philosophy.md](./philosophy.md) の優先順位と同じ）。

1. 作品と作家が主役に見えるか
2. 完成形としてイラストレーター優勢に見えるか
3. 既存の public pages / Prisma schema / License モデルを壊さないか
4. Prompt 6 を MVP として小さく閉じられるか
5. Prompt 5 までの導線と自然につながるか
